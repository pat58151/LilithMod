Add LLM-backed free-text chat to the existing LilithMod BepInEx plugin (step 3). The player
opens a text box, types anything, and Lilith replies in her own dialogue bubble using a
response generated by a remote LLM. Extend the existing plugin; do not rewrite steps 1-2.

## Architecture (fixed by design - implement exactly this)

**Output path - reuse proven machinery, do NOT invent new rendering.**
Reserve a single dedicated DialogueNode with id 9500000, injected at startup exactly like
step 2's custom nodes (lineId = 0, inline text, no trigger, speaker "lilith"). When a reply
arrives, set that node's `text` to the reply string and call
`DialogueManager.s_instance.StartDialogue(9500000)`. This path is already verified working:
inline text with lineId 0 renders correctly in the game's own bubble.

**Threading - this is the hard requirement.**
Unity/IL2CPP will crash if game objects are touched off the main thread. Therefore:
- The HTTP request runs on a background thread (Task + HttpClient).
- The background thread MUST NOT touch any Unity or Il2Cpp object. It produces a plain
  C# string and pushes it onto a thread-safe queue (e.g. ConcurrentQueue<string>).
- A MonoBehaviour `Update()` on the main thread drains that queue and does all game
  interaction (setting node text, StartDialogue, UI updates).
- The game must never block or freeze while a request is in flight.

**Input UI.**
Build a uGUI text input: a Canvas (screen space overlay, high sortingOrder) containing a
TMPro.TMP_InputField. Toggle visibility with a configurable hotkey. Enter submits, Escape
closes. uGUI is known to work in this game (its own choice buttons are uGUI and respond to
clicks), so reuse the EXISTING EventSystem found via Object.FindObjectOfType rather than
creating a second one - two EventSystems break input.
CRITICAL: the field must use one of the game's own TMP fonts or Chinese text renders as
blank boxes. Obtain a TMP_FontAsset from a live game text component at runtime (for example
the `font` of an existing TextMeshProUGUI in the scene) rather than relying on the TMP
default, which is Latin-only.

**Config** (BepInEx ConfigFile, section "LLM"):
`BaseUrl` (default "https://api.deepseek.com/v1"), `ApiKey` (default ""), `Model`
(default "deepseek-chat"), `SystemPrompt`, `MaxHistoryTurns` (default 8),
`TimeoutSeconds` (default 30), `Hotkey`. The request is an OpenAI-compatible
POST to {BaseUrl}/chat/completions with Bearer auth, so any compatible provider works.
NEVER hardcode or commit a key.

**Persona and history.**
Maintain a rolling conversation history capped at MaxHistoryTurns exchanges. Send it with
each request. The system prompt must establish Lilith's character; seed the default with a
handful of her real lines taken from the dumped dialogue so replies match her voice.

## Hard constraints
- Reuse the existing Update()-polling attach pattern; do NOT create your own GameObject in
  Load() (it runs before the first scene and never ticks). Use BasePlugin.AddComponent.
- Injected MonoBehaviours have no `Log`; log through `LilithModPlugin.Logger`.
- Il2Cpp collection fields need Il2CppSystem collections, not System.Collections.Generic.
- No exception may escape into the game, on any path, including background threads.
  An unobserved Task exception must be caught and logged.

## Definition of done
- `dotnet build -c Release` succeeds with zero errors.
- Pressing the hotkey in-game opens a text box that accepts typed input and shows it.
- Submitting text produces a reply from the configured LLM in Lilith's normal dialogue
  bubble, and the log records the request being sent and a reply being received.
- The game never freezes while waiting: it keeps animating and responding during the call.
- With an empty ApiKey, an unreachable BaseUrl, or a timeout, the plugin logs a clear
  warning, shows a graceful in-character fallback line instead of the reply, and the game
  continues running normally with vanilla behaviour intact.
