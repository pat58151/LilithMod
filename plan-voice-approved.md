## Implementation Plan: TTS Voice Output for Lilith

### 1. Overview

Extend LilithMod so that chat replies generated by `LlmChatController` are synthesised to speech by a local GPT‑SoVITS HTTP service and played through NAudio. The feature must be optional, fault‑tolerant, and off‑thread. Warm‑up logic shields the user from first‑use compilation latency.

**Outcome**  
- Every reply text still appears in the chat UI exactly as before.  
- When voice is enabled and the service is reachable, the reply is also spoken aloud.  
- If the service is unavailable, the system logs one warning and continues silently.  
- All synthesis and playback happen on a background thread; no per‑frame work is added to `Update()` beyond a lightweight queue‑drain check if needed (the existing queue style is acceptable).

---

### 2. Architecture

```
Config (BepInEx) → VoiceConfig (object)
                     ↓
LlmChatController.HandleChatResult
  → if Voice enabled → SpeechQueue.Enqueue(text)
                     ↓
         SpeechQueueProcessor (dedicated thread)
           ↓ waits on _warmUpComplete signal
           ↓ loop:
           │   text ← queue.Dequeue()
           │   wavBytes ← TtsClient.SynthesizeAsync(text)   // HTTP POST
           │   VoicePlayer.PlaySync(wavBytes)                // blocking playback
           ↓
         TtsClient → HTTP to http://127.0.0.1:9880/tts
         VoicePlayer → NAudio WaveOutEvent, waits for PlaybackStopped
```

Startup sequence:

1. Read `VoiceConfig` from config file.
2. If `VoiceEnabled` is `true`, create a `SpeechQueueProcessor`, a `TtsClient`, and start a `WarmUpTask`.
3. `WarmUpTask` sends several fixed sentences (~20‑30 chars) sequentially, with a total timeout. When all succeed or the timeout expires, it sets `_warmUpComplete` signal. Any failure is logged but does **not** block the signal.
4. `SpeechQueueProcessor`’s main loop waits for `_warmUpComplete` before draining the queue. This guarantees the first real reply is never synthesised on a cold service.

### 3. Detailed Components

#### 3.1 VoiceConfig

Read from `BepInEx/config/LilithMod.cfg` under a new `[Voice]` section.

| Key                  | Type   | Default                                    | Description                                      |
| -------------------- | ------ | ------------------------------------------ | ------------------------------------------------ |
| `Enabled`            | bool   | `false`                                    | Master switch.                                   |
| `Endpoint`           | string | `http://127.0.0.1:9880/tts`               | Full URL of the TTS service.                     |
| `RefAudioPath`       | string | `<modRoot>/voice/jp/calm-reference.wav`    | Absolute path to the reference audio on disk.    |
| `PromptText`         | string | `Transcript_of_the_reference_clip`         | Exact transcript for `prompt_text` field.        |
| `TextLang`           | string | `ja`                                       | Language of the input text.                      |
| `PromptLang`         | string | `ja`                                       | Language of the reference clip.                  |
| `TimeoutSeconds`     | int    | 60                                         | HTTP request timeout (seconds).                  |
| `WarmUpTimeoutSeconds`| int   | 120                                        | Total warm‑up time budget (seconds).             |
| `FragmentInterval`   | float  | 0.4                                        | `fragment_interval` for service.                 |
| `TextSplitMethod`    | string | `cut5`                                     | `text_split_method`.                             |

`RefAudioPath` is converted to an absolute path at runtime using the mod’s installation directory.

#### 3.2 TtsClient

- Uses `System.Net.Http.HttpClient` (one static instance) to `POST` to the configured endpoint.
- **Request body (JSON)** – schema exactly matching the verified service contract:

  ```json
  {
    "text": "<the reply text>",
    "text_lang": "<TextLang>",
    "ref_audio_path": "<absolute RefAudioPath>",
    "prompt_text": "<PromptText>",
    "prompt_lang": "<PromptLang>",
    "media_type": "wav",
    "streaming_mode": false,
    "text_split_method": "<TextSplitMethod>",
    "fragment_interval": <FragmentInterval>
  }
  ```

- All fields are mandatory and provided by `VoiceConfig`.  
- **Response handling**:
  - Success: HTTP 200 with `Content‑Type: audio/wav` → return the raw bytes as `byte[]`.
  - Error: HTTP 400 with JSON error body `{"message":..., "Exception":...}` → throw an exception containing the message.
  - Network failure / timeout → throw `TimeoutException` or `HttpRequestException`.
- **Timeout**: The HTTP client is configured with `Timeout` from config.
- **Serialization**: Use **Newtonsoft.Json** (`JsonConvert.SerializeObject`) for the request body. No Base64 anywhere.

#### 3.3 VoicePlayer

- Takes raw WAV bytes and plays them synchronously using **NAudio**.
- Implementation:
  - Create a `MemoryStream` from the byte array.
  - Load it as a `WaveFileReader` (handles the WAV header and PCM data).
  - Use `WaveOutEvent` (not `WaveOut`) because it runs on a dedicated playback thread and is safe to call from a background thread.
  - Subscribe to `PlaybackStopped` event.
  - Call `Play()`, then block until `PlaybackStopped` fires. A blocking wait can use a `ManualResetEventSlim` that is set inside the event handler.
  - After completion, dispose all NAudio objects.
- **Blocking behaviour**: The method does not return until audio is fully played. This ensures a new utterance never cuts off the current one – the queue processor is blocked until playback finishes.

#### 3.4 SpeechQueue

- Simple `System.Collections.Concurrent.ConcurrentQueue<string>`.
- Enqueued by `LlmChatController` when voice is enabled.
- Draink by `SpeechQueueProcessor`.

#### 3.5 SpeechQueueProcessor

- A **dedicated background thread** (created with `new Thread() { IsBackground = true }` or `Task.Factory.StartNew(..., TaskCreationOptions.LongRunning)`).
- **Startup dependency**: It waits on a `ManualResetEventSlim _warmUpComplete` before entering its main loop.
- **Main loop** (pseudo‑code):

  ```
  while (!_cancellationToken.IsCancellationRequested) {
      if (queue.TryDequeue(out string text)) {
          try {
              byte[] wav = ttsClient.SynthesizeAsync(text).Result; // synchronous wait on this thread
              voicePlayer.PlaySync(wav);
          } catch (Exception ex) {
              Log.Warning($"Speech failed: {ex.Message}");
              // continue – never break chat
          }
      } else {
          Thread.Sleep(100); // idle when queue empty
      }
  }
  ```

- **Thread‑safety**: The queue is concurrent; all NAudio calls are thread‑safe because `WaveOutEvent` is used.
- On shutdown, cancel the token and join the thread.

#### 3.6 WarmUpTask

- Runs on a **separate short‑lived background thread**.
- Defines a hardcoded array of warm‑up sentences, each **15‑30 characters** long, representative of Lilith’s typical replies. Examples:
  - "ちょっと待って、今調整しているところだ。"
  - "なるほど、面白い質問ですね。少し考えさせてください。"

- For each sentence:
  1. Call `ttsClient.SynthesizeAsync(sentence).Result`. If it throws, log the exception but **continue** to the next sentence (do not abort warm‑up).
  2. Do **not** play the resulting audio; discard it.
- Track the total elapsed time with a `Stopwatch`. If the warm‑up loop exceeds `WarmUpTimeoutSeconds`, break out and log a warning.
- After the loop ends (all sentences processed or timeout), **set `_warmUpComplete`**.
- No real syntheses from `HandleChatResult` are processed before `_warmUpComplete` is set, because the processor thread is blocked there.

**Note**: The one‑off kernel compilation cost is per input‑length shape; using 2‑3 medium‑length sentences effectively warms the model for the lengths that real replies will have.

#### 3.7 Integration with LlmChatController

- In `HandleChatResult`, after the existing text display logic, add:

  ```
  if (VoiceConfig.Enabled) {
      SpeechQueue.Enqueue(replyText);
  }
  ```

- The text path remains unchanged.

#### 3.8 Startup & Shutdown

- In the mod’s `Awake()` (or equivalent setup method):
  1. Read `VoiceConfig` from BepInEx `Config` (already available).
  2. If `Enabled` is `false`, skip all voice initialisation.
  3. Build absolute path for `RefAudioPath` (e.g., `Path.Combine(modDirectory, "voice", "jp", "calm-reference.wav")`). Verify the file exists; if missing, log an error and disable voice with a single warning.
  4. Instantiate `TtsClient`, `VoicePlayer`, `SpeechQueueProcessor`, and `WarmUpTask` (passing the warm‑up sentences and the signal).
  5. Start the warm‑up thread, then start the speech processor thread.
- `OnDestroy()`: Cancel the processor’s token, signal the playback waiter if needed, and dispose NAudio resources.

#### 3.9 Project changes (`LilithMod.csproj`)

Add the NAudio package:

```xml
<ItemGroup>
  <PackageReference Include="NAudio" Version="2.2.1" />
  <!-- existing references -->
</ItemGroup>
```

Ensure `CopyLocalLockFileAssemblies` is already set to `true` (it is) so the NAudio DLL is copied to the plugin output.

**No other dependencies** – NAudio is managed .NET and works in IL2CPP without native libraries.

### 4. Error Handling & Robustness

- **All TTS exceptions are caught inside the `SpeechQueueProcessor` loop** and logged as warnings. The queue continues processing subsequent messages. The chat UI is never affected.
- **Service absent**: First HTTP request fails → warning logged. Subsequent attempts will also fail; each failure produces a single log line (not a storm). An optional internal back‑off can be added, but not required.
- **Invalid response (e.g., 400)**: The error message from the service JSON is logged and that specific utterance is skipped.
- **Playback failure**: NAudio exceptions are caught and logged; the processor continues.
- **Warm‑up failure**: If the service never becomes available during warm‑up, the signal is still set after timeout, allowing real speech attempts (which will likely fail, but gracefully). The first real reply will then be cold – this is acceptable because we cannot force the service to warm up if unreachable.
- **No new per‑frame overhead** – the only interaction from Unity’s main thread is the one‑line enqueue call and checking the `_warmUpComplete` signal during startup (which is done once).

### 5. Verification & Acceptance Criteria

#### 5.1 Behavioural Acceptance Criteria

| # | Criterion | Verification |
| - | --------- | ------------ |
| AC1 | With TTS service running: chat replies produce audible speech. The text still appears in chat exactly as before. | Start service, start mod, send a chat message; listen for spoken reply and confirm text in UI. |
| AC2 | With TTS service **not** reachable: chat works normally, no exception, no hang. The log contains a single warning message (e.g., “Speech synthesis failed…”) per failed attempt. | Stop service, restart mod, chat; verify UI no freeze, check BepInEx console for warning, no error. |
| AC3 | Voice is disabled (config `Enabled = false`): no speech is attempted, no network calls, no background threads for TTS are started. | Set false, chat; check that no HTTP traffic occurs, log no voice‑related messages. |
| AC4 | The first real reply after a fresh mod load (service already warm) is not delayed by kernel compilation – a warm‑up phase runs at startup. | After mod loads, wait ~30 seconds before sending a message; reply should speak immediately without a multi‑second pause. |
| AC5 | A new reply that arrives while a previous reply is still playing is queued and played **after** the current audio finishes. Never cut off. | Send two messages in quick succession; hear both replies sequentially with no overlap or truncation. |
| AC6 | All synthesis and playback run off the Unity main thread. | Use Unity Profiler or a manual log of `Thread.CurrentThread.ManagedThreadId` to show playback occurs on a different thread than the main thread (captured in `Awake`). |
| AC7 | `ref_audio_path`, `text_lang`, `prompt_lang`, `text_split_method`, `fragment_interval` are all sent in the request; `base64` is never used. | Inspect HTTP traffic or add temporary log of request body; must contain those fields and no `audio` field. |
| AC8 | The configuration file is complete with all needed keys and sensible defaults, including `RefAudioPath` pointing to the shipped `jp/calm-reference.wav`. | Open config file, verify section and keys. |
| AC9 | The project builds successfully and includes NAudio. | Run `dotnet build`; examine output directory for `NAudio.dll`. |

#### 5.2 Automated Verification Command

A Python script (`verify-voice.py`) placed in the repository root, mirroring the style of `verify-step3.py`, is required. It must:

1. Confirm the mod builds:  
   `subprocess.run(["dotnet", "build", "-c", "Release"], cwd="LilithMod")` – expect exit code 0.
2. Check essential source elements exist using `grep` (or equivalent file‑search for Windows compatibility – just use `Path` and file content reading):
   - In `LilithMod.csproj`: the string `NAudio`.
   - In the C# source files (any `.cs` under the mod folder):
     - `ref_audio_path` (exact case‑insensitive)
     - `text_lang`
     - `fragment_interval`
     - `PlaybackStopped`
     - `Newtonsoft.Json`
   - Verify that no file contains `base64` in the TTS‑related code (can be a simple false‑positive check; enough to grep `base64` – if found, warn but not fail because it might occur elsewhere; we will commit to not using it, so it should be absent in new code).
3. Confirm the output binary `LilithMod/Plugins/LilithMod.dll` exists.
4. Confirm that `LilithMod/Plugins/NAudio.dll` (or a matching NAudio assembly) exists after build.

Run: `python verify-voice.py` – exits 0 on success, non‑zero with clear error messages otherwise.

**Note**: The build must succeed with the NAudio package in a netstandard2.1 IL2CPP environment; the script does not check runtime behaviour, only compile‑time and packaging integrity.

### 6. Implementation Steps (ordered)

1. **Edit `LilithMod.csproj`** – add NAudio package reference.
2. **Create `VoiceConfig.cs`** – read all configuration values from BepInEx `Config`, handle path conversion.
3. **Create `TtsClient.cs`** – with `HttpClient`, Newtonsoft.Json serialization of request, error parsing, timeout.
4. **Create `VoicePlayer.cs`** – synchronous WAV playback using NAudio (`WaveFileReader`, `WaveOutEvent`, manual reset event).
5. **Create `SpeechQueueProcessor.cs`** – background thread, queue drain loop, warm‑up gate.
6. **Create `WarmUpTask.cs`** – sends warm‑up sentences, sets signal; runs on separate thread.
7. **Modify `LlmChatController.cs`** – enqueue text if voice enabled.
8. **Modify mod entry point** – setup all voice components on startup, disposal on destroy.
9. **Test** against acceptance criteria manually.
10. **Create `verify-voice.py`** and run it.

---

All files must reside in the `LilithMod` source tree. No changes to existing config infrastructure except adding the new section.
